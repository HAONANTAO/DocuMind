## 1. 项目概述

### 1.1 背景

大型语言模型（LLM）的训练数据存在截止日期，无法感知用户的私有文档内容。企业员工在查阅内部文档时，往往需要手动翻找，效率低下。DocuMind 通过 RAG（检索增强生成）技术，让用户以自然语言提问的方式精准获取文档中的信息。

### 1.2 产品定位

> "上传你的文档，像问人一样问问题。"

DocuMind 是一款基于 RAG 技术的私有文档问答 SaaS，用户注册后可上传 PDF 文档，通过对话界面进行多轮提问，AI 将基于文档内容作答并标注来源页码。

### 1.3 目标用户

| 用户类型 | 典型场景 |
|----------|----------|
| 企业新员工 | 快速检索员工手册、公司制度等内部文档 |
| 法务 / 合规人员 | 对合同、政策文件进行快速条款查询 |
| 学生 / 研究者 | 对论文、报告进行内容提问与摘要 |

### 1.4 核心价值主张

- **精准**：AI 仅基于上传文档内容回答，不凭空编造
- **透明**：每条回答标注原文来源（页码 + 片段）
- **便捷**：无需学习成本，像聊天一样使用
- **安全**：每位用户数据严格隔离，他人无法访问

## 2. 商业模式（SaaS）

### 2.1 定价方案

| 套餐 | 价格 | 文档数量上限 | 每月提问次数 |
|------|------|------------|--------------|
| Free | $0 / 月 | 3 个 | 50 次 |
| Pro | $19 / 月 | 无限 | 无限 |
| Enterprise | 联系销售 | 无限 + 私有部署 | 无限 |

> MVP 阶段定价页静态展示即可，无需真实接入支付。

### 2.2 转化策略

- Free 套餐作为用户获取入口
- 核心转化点：触达文档数量或提问次数上限时，引导用户升级 Pro
- 后续迭代可接入 Stripe 实现真实订阅

## 3. 功能需求

### 3.1 优先级定义

| 优先级 | 说明 |
|--------|------|
| P0 | 核心功能，MVP 必须交付 |
| P1 | 重要功能，第一版尽量覆盖 |
| P2 | 加分功能，有余力再做 |

---

### 3.2 用户认证（P0）

- 用户注册（邮箱 + 密码）
- 用户登录 / 登出
- JWT Token 鉴权，所有接口需携带 Token
- 密码加密存储（bcrypt）

---

### 3.3 文档管理（P0）

- 支持上传 PDF 格式文件，单文件上限 20MB
- 上传后自动触发解析流程：切片 → Embedding → 写入向量库
- 文档库页展示所有已上传文档（名称、上传时间、解析状态）
- 支持删除文档，同步清除向量库中对应数据

**文档解析状态枚举：**

| 状态 | 说明 |
|------|------|
| uploading | 文件上传中 |
| processing | 正在解析 / 向量化 |
| ready | 解析完成，可提问 |
| error | 解析失败 |

---

### 3.4 文档问答（P0）

- 针对已上传文档发起对话提问
- 支持多轮对话（携带上下文历史）
- AI 回答必须附带来源引用（页码 + 原文片段）
- 对话历史持久化存储，刷新后不丢失
- AI 回答采用流式输出（SSE），前端展示打字机效果

**Prompt 策略（核心）：**

```
你是一个文档问答助手。
请仅根据以下文档片段回答用户问题，不要使用文档以外的知识。
如果文档中没有相关信息，请直接说"文档中未找到相关内容"。
回答结束后，列出你引用的原文片段及其页码。

[文档片段]
{retrieved_chunks}

[用户问题]
{user_question}
```

---

### 3.5 定价页（P1）

- 静态展示 Free / Pro / Enterprise 三档套餐
- 每档列出功能对比
- Pro 套餐按钮点击后提示"敬请期待"（MVP 阶段）

---

### 3.6 用量限制提示（P1）

- Free 用户文档超过 3 个时，上传按钮灰化并提示升级
- Free 用户提问超过 50 次时，输入框禁用并展示升级弹窗

---

### 3.7 加分功能（P2）

- 对话内容导出为 PDF
- 文档解析实时进度条
- 回答质量反馈（👍 / 👎）

## 4. 非功能需求

| 类型 | 要求 |

| 性能 | 普通提问响应时间 ≤ 5 秒（非流式模式） |

## 5. 技术架构

### 5.1 技术栈选型

| 层级 | 技术选型 | 选型理由 |
|------|----------|----------|
| 前端 | React + Tailwind CSS | 与现有 MERN 技能匹配 |
| 后端 | Node.js + Express | 与现有 MERN 技能匹配 |
| 业务数据库 | MongoDB | 存用户信息、对话历史 |
| 向量数据库 | Pinecone | 免费层够用，API 简单 |
| AI 框架 | LangChain.js | 封装 RAG 链路，上手快 |
| AI 模型 | OpenAI GPT-4o-mini | 性价比高，适合开发调试 |
| 部署（前端） | Vercel | 免费，自动 CI/CD |
| 部署（后端） | Railway | 免费层支持 Node.js |

---

### 5.2 RAG 核心流程

```
[用户上传 PDF]
      │
      ▼
[PDF 解析]
文本提取，按 1000 字符切片，片段间 200 字符重叠
      │
      ▼
[Embedding]
调用 OpenAI text-embedding-3-small
将每个片段转为 1536 维向量
      │
      ▼
[向量存储]
存入 Pinecone
元数据携带：documentId、userId、pageNumber、原文片段
      │
      ▼
[用户提问]
      │
      ▼
[语义检索]
将问题向量化
从 Pinecone 检索 Top-5 相关片段（按 userId + documentId 过滤）
      │
      ▼
[构建 Prompt]
检索结果 + 对话历史 + 用户问题 → 组合成 Prompt
      │
      ▼
[LLM 生成]
调用 GPT-4o-mini，流式返回
      │
      ▼
[返回答案 + 来源引用]
存入 MongoDB 对话历史
```

## 7. 交付计划（4 周）

### 第 1 周：跑通 AI 链路

- 注册 OpenAI、Pinecone 账号，获取 API Key
- 搭建 Node.js 项目基础结构
- 实现 PDF 解析 → 切片 → Embedding → 存入 Pinecone
- 实现提问 → 向量检索 → 构建 Prompt → 返回答案
- **目标：命令行里跑通完整 RAG 流程，不需要界面**

### 第 2 周：后端 API

- 用户注册 / 登录（JWT + bcrypt）
- 文档上传接口（multer + 异步解析）
- 对话接口（接入第 1 周的 RAG 链路）
- MongoDB 存储对话历史
- **目标：用 Postman 能跑通所有接口**

### 第 3 周：前端界面

- 登录 / 注册页
- 文档库页
- 对话页（含流式打字机效果）
- 定价页
- **目标：前后端联调完成，本地完整可用**

### 第 4 周：上线 + 打磨

- 部署前端到 Vercel，后端到 Railway
- 写 README（含架构图、本地运行说明）
